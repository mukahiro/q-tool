# アーキテクチャとチーム開発の方針

本プロジェクトは初心者を含むチームでの開発を想定し、**「わかりやすさ」と「分業のしやすさ」** を最優先としたアーキテクチャを採用します。

## 1. ディレクトリ構成：機能（Feature）でまとめる
「画面の見た目」や「裏側の処理」といった技術的な種類で分けるのではなく、**「アプリの機能ごと」** にファイルを整理します。これにより、「どこに何を書けばいいか」が直感的にわかりやすくなります。

```text
src/
├── app/               # 画面のURLを決めるだけの薄い層
│   ├── room/[id]/page.tsx
│   └── globals.css
├── features/          # アプリの「機能」ごとにまとめる（ここがメインの開発場所）
│   ├── room/          # ルーム機能（入室、QRコードなど）
│   ├── question/      # 質問機能（投稿、一覧表示、いいねなど）
│   │   ├── components/  # 質問機能専用のUI部品
│   │   └── actions.ts   # 質問データを保存・更新する処理
│   └── summary/       # AI要約機能
├── components/        # 色々な機能で使い回す共通のUI部品（ボタンなど）
└── lib/               # 便利ツールや外部設定（データベース接続など）
```

## 2. ビュー（画面）の書き分け

画面の見た目（ビュー）は、役割に応じて `src/app` と `src/features` に書き分けます。これは、**「app は配置図、features は家具（部品）」** というイメージです。

### A. `src/app` に書くこと（画面の枠組み）
URL（ページ）ごとの全体のレイアウトや、どの機能をどこに置くかを決めます。
- 例：`/room/[id]/page.tsx`
- ここでは具体的なボタンやリストの見た目は作らず、`features` から部品を呼び出して配置するだけにします（薄いページ）。

### B. `src/features/.../components` に書くこと（具体的な部品）
その機能の「中身」となる具体的な見た目を作ります。
- 例：質問リスト、投稿フォーム、要約カードなど。
- **実際のUI開発（CSSの調整など）の大部分はここで行います。**
- これにより、一つの部品を別のページでも再利用しやすくなります。

## 3. エンティティ（データ）と処理（ロジック）の分け方

従来の「オブジェクト指向（クラス）」のような複雑な設計は避け、**「データ（型）」と「処理（関数）」を分けて考える** シンプルなルールで開発します。

### A. データは Prisma に任せる
データベースのテーブル構造から、Prismaが自動的に TypeScriptの「型（Type）」を作ってくれます。チーム全員がこの**「Prismaが作った型」をエンティティの正解として使い回します。**

*例：質問のデータを画面に表示する時*
```typescript
import type { Question } from '@prisma/client' // Prismaが作った型を使う

export function QuestionItem({ question }: { question: Question }) {
  return <div>{question.content}</div>;
}
```

### B. データの保存・更新は「関数」にする (Server Actions)
「質問を保存する」「いいねを押す」といった処理は、クラスのメソッドではなく、独立した関数として `actions.ts` にまとめます。

*例：いいねを押す処理*
```typescript
// features/question/actions.ts
"use server"

export async function addLike(questionId: string) {
  // データベースを更新する処理を書く
}
```

## 3. チーム開発のルールまとめ

1.  **迷ったら機能（`features/`）に入れる:** その画面や機能でしか使わない部品や処理は、すべて `features/` の下の関連するフォルダに入れます。
2.  **型は自作せず Prisma を頼る:** データベースに関連するデータの型は、自分で新しく作らずに `@prisma/client` からインポートして使います。
3.  **UIとデータ保存を分ける:** 画面の見た目を作るファイル (`components/`) と、データを書き換えるファイル (`actions.ts`) を分けることで、複数人での作業がぶつかりにくくなります。
