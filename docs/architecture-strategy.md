# アーキテクチャとチーム開発の方針

本プロジェクトは初心者を含むチームでの開発を想定し、**「わかりやすさ」と「分業のしやすさ」** を最優先としたアーキテクチャを採用します。

## 1. ディレクトリ構成：機能（Feature）でまとめる
「画面の見た目」や「裏側の処理」といった技術的な種類で分けるのではなく、**「アプリの機能ごと」** にファイルを整理します。これにより、「どこに何を書けばいいか」が直感的にわかりやすくなります。

```text
src/
├── app/               # 画面のURLを決めるだけの薄い層
│   ├── rooms/[roomId]/page.tsx
│   ├── join/[inviteCode]/page.tsx
│   ├── rooms/[roomId]/chat/page.tsx
│   └── globals.css
├── features/          # アプリの「機能」ごとにまとめる（ここがメインの開発場所）
│   ├── room/          # ルーム機能（入室、QRコードなど）
│   │   ├── components/  # ルーム機能専用のUI部品
│   │   ├── hooks/       # ルーム画面で使う状態管理・画面用ロジック
│   │   ├── utils/       # ルーム機能内で使う小さな変換・整形処理
│   │   └── actions.ts   # ルームデータを保存・更新するサーバー処理
│   ├── question/      # 質問機能（投稿、一覧表示、いいねなど）
│   │   ├── components/  # 質問機能専用のUI部品
│   │   ├── hooks/       # 質問画面で使う状態管理・画面用ロジック
│   │   ├── utils/       # 質問機能内で使う小さな変換・整形処理
│   │   └── actions.ts   # 質問データを保存・更新するサーバー処理
│   ├── auth/          # 認証機能（ログイン、ログアウトなど）
│   │   ├── components/  # 認証機能専用のUI部品
│   │   ├── hooks/       # 認証画面で使う状態管理・画面用ロジック
│   │   ├── utils/       # Firebaseエラー変換などの小さな処理
│   │   └── actions.ts   # Cookie保存などのサーバー処理
│   └── summary/       # AI要約機能
├── components/        # 色々な機能で使い回す共通のUI部品（ボタンなど）
└── lib/               # 便利ツールや外部設定（データベース接続など）
```

## 2. ビュー（画面）の書き分け

画面の見た目（ビュー）は、役割に応じて `src/app` と `src/features` に書き分けます。これは、**「app は配置図、features は家具（部品）」** というイメージです。

### A. `src/app` に書くこと（画面の枠組み）
URL（ページ）ごとの全体のレイアウトや、どの機能をどこに置くかを決めます。
- 例：`/rooms/[roomId]/page.tsx`
- ここでは具体的なボタンやリストの見た目は作らず、`features` から部品を呼び出して配置するだけにします（薄いページ）。

### B. `src/features/.../components` に書くこと（具体的な部品）
その機能の「中身」となる具体的な見た目を作ります。
- 例：質問リスト、投稿フォーム、要約カードなど。
- **実際のUI開発（CSSの調整など）の大部分はここで行います。**
- これにより、一つの部品を別のページでも再利用しやすくなります。

## 3. エンティティ（データ）と処理（ロジック）の分け方

従来の「オブジェクト指向（クラス）」のような複雑な設計は避け、**「データ（型）」と「処理（関数）」を分けて考える** シンプルなルールで開発します。

### A. データ型は機能ごとに定義する
Firestore は NoSQL データベースであり、Prisma のようにスキーマから自動生成される型は使いません。チーム全員が同じ形でデータを扱えるように、必要な TypeScript 型を各機能フォルダ内で定義します。

*例：質問のデータを画面に表示する時*
```typescript
type Question = {
  id: string;
  sectionId: string;
  content: string;
  studentSessionId: string;
  createdAt: Date;
};

export function QuestionItem({ question }: { question: Question }) {
  return <div>{question.content}</div>;
}
```

### B. データの保存・更新は「関数」にする (Server Actions)
「質問を保存する」「いいねを押す」といった処理は、クラスのメソッドではなく、独立した関数として `actions.ts` にまとめます。Firestore への読み書きも、基本的には該当機能の `actions.ts` または専用のヘルパー関数に閉じ込めます。

*例：いいねを押す処理*
```typescript
// features/question/actions.ts
"use server"

export async function addLike(questionId: string) {
  // データベースを更新する処理を書く
}
```

## 4. 分けすぎない分離の方針

初心者が多いチームでは、最初から細かく分けすぎると全体像が追いにくくなります。一方で、すべてを1ファイルに詰め込むと後から読みにくくなります。そのため、本プロジェクトでは **「見た目」「画面用ロジック」「サーバー処理」「小さな変換処理」** を必要に応じて分けます。

- **components/**: 見た目を中心に書く。短いクリック処理や `isLoading` 程度の状態管理は残してよい。
- **hooks/**: 入力状態、画面内の処理手順、Firebase のログイン処理など、コンポーネントが長くなったときに切り出す。
- **actions.ts**: Cookie 操作、DB更新、サーバーでしかできない処理を書く。`"use server"` を付ける。
- **utils/**: エラー文言の変換、日付や文字列の整形など、画面に依存しない小さな処理を書く。

目安として、UIファイルが100〜150行を超えたり、Firebase/Firestore処理や `switch` 文が目立ってきたら切り出します。逆に、1つのボタン専用の短い処理まで無理に分ける必要はありません。

## 5. チーム開発のルールまとめ

1.  **迷ったら機能（`features/`）に入れる:** その画面や機能でしか使わない部品や処理は、すべて `features/` の下の関連するフォルダに入れます。
2.  **型は機能ごとに揃える:** Firestore のドキュメント形状に対応する TypeScript 型を、関連する `features/` 内に定義して使います。
3.  **UIとデータ保存を分ける:** 画面の見た目を作るファイル (`components/`) と、データを書き換えるファイル (`actions.ts`) を分けることで、複数人での作業がぶつかりにくくなります。
