# 技術スタックの検討と決定

本プロジェクトで採用する技術スタックとその選定理由をまとめます。

## フロントエンド・フレームワーク

### **Next.js (App Router)**
- **選定理由:** Reactベースの最新フレームワークであり、サーバーコンポーネント (RSC) による高いパフォーマンスと、Vercelへのデプロイの親和性を考慮しました。
- **ルーティング:** ファイルベースのApp Routerを使用し、モダンな開発エクスペリエンスを優先します。

## 開発言語

### **TypeScript**
- **選定理由:** 静的型付けによるバグの早期発見、コードの自己文書化、および大規模開発における保守性の向上を目的としています。

## スタイリング

### **Tailwind CSS**
- **選定理由:** ユーティリティファーストな設計により、スタイル定義の高速化とデザインの一貫性を保ちます。Next.jsとの相性も良く、標準的な構成として採用しました。

## データベース

### **Firebase Firestore**
- **選定理由:** 
  - Firebase Authentication と同じ Firebase プロジェクト内で管理でき、認証情報とデータ権限を連携しやすい。
  - リアルタイム同期に強く、学生の質問投稿やリアクションの反映に向いている。
  - サーバー管理が不要で、開発初期から素早く導入できる。
- **データモデル:** Firestore のコレクション / ドキュメント構造で管理します。PostgreSQL のような外部キー制約はないため、参照先IDを保存し、アプリケーション側とセキュリティルールで整合性を守ります。

## 認証

### **Firebase Authentication**
- **選定理由:** 教師向けのメールアドレス・パスワード認証を素早く導入でき、クライアント側の認証状態監視と Next.js の Server Action を組み合わせやすいため採用しました。
- **実装方針:** Firebase の ID トークンを httpOnly Cookie に同期し、サーバー側のページでもログイン状態を判定できるようにします。

## データアクセス

### **Firebase SDK**
- **選定理由:** 
  - Firestore は NoSQL データベースのため、Prisma のような ORM は使用しません。
  - クライアント側では Firebase Web SDK、サーバー側の管理処理では必要に応じて Firebase Admin SDK を使用します。
  - TypeScript の型は、機能ごとに必要なデータ型を定義して利用します。

## AI API

### **Google AI Studio (Gemini API)**
- **選定理由:** 高性能な言語モデル Gemini を利用でき、開発者向けの無料枠や使いやすさを考慮して採用しました。質問の要約やカテゴリ分けに使用します。

## インフラ・デプロイ

### **Vercel**
- **選定理由:** Next.jsの開発元であり、ゼロ構成でデプロイ・プレビュー環境の構築が可能。サーバーレス環境との統合も最適化されています。
